Step by Step: Speeding Up JDBC Access

By Jason Price

Disabling auto-commit and using update batching.

As the first Java application-programming interface (API) to provide connectivity to databases, Java Database Connectivity (JDBC) is one of the first technologies Java developers become exposed to when creating applications for Oracle and other databases. In this article, you can learn about two basic JDBC techniques that can significantly improve application performance. Better yet, by downloading, compiling, and running the accompanying sample code, you can see for yourself how these techniques improve performance.

Before running the samples, you must create a performance table for the test data. I've created this table in the familiar scott schema, so if you use a different schema, be sure to change the schema name in the sample applications. You may also need to change the name of the Oracle database service—mine uses the default ORCL service name and runs locally. (Although it's not good practice to hard-code the database connection-string information, as I've done in these samples, it's OK for test purposes.)

Use this SQL statement to create the performance table:

CREATE TABLE performance (
  int_value INTEGER,
  varchar2_value VARCHAR2(10)
);

As you can see, the performance table contains an INTEGER column (int_value) and a VARCHAR2 column (varchar2_value). These columns will be filled with test data generated by the sample programs; the programs also capture the start and the end times and calculate the difference between the two for each run.

Let's start by disabling the auto-commit mode.

What Auto-Commit Does, and Why You Should Disable It

By default, the auto-commit mode in JDBC is enabled; that is, every SQL statement sent to the database is automatically committed to the database. Committed to the database means that the change is permanently recorded in the database after the SQL statement INSERT, UPDATE, or DELETE has executed. From an application-design perspective, it rarely makes sense to keep auto-commit mode enabled, because at the business-logic level of the application, transactions frequently comprise multiple SQL statements, not just single INSERT, UPDATE, or DELETE statements. As the developer, you must think about how your transactions should be demarcated and what SQL statements they should contain.

For example, a transaction might consist of two UPDATE statements that effect a funds transfer from one account to another. This transfer might use one UPDATE statement to subtract money from a checking account and another to add money to a savings account. Because of the potential for losing money if an error occurs in either of the UPDATE statements, both must be made permanent by using a commit or they must both be undone by using a rollback.

If you were to leave auto-commit enabled, each statement would be committed as it executes, potentially leaving the database in an inconsistent state. In my previous example, money could be lost. (See the "ACID Transaction Properties" sidebar in the article "JDBC Transaction Optimization," in the Oracle Developer section of Oracle Magazine and "Using Database Transactions in Java" for more information about transactions and the consistency requirement.)

As you can probably tell, it makes no sense to leave auto-commit enabled when the typical transaction defined in an application comprises more than one SQL statement. In addition, disabling auto-commit results in a performance benefit.

Before disabling auto-commit for your java.sql.Connection (or oracle.jdbc.OracleConnection) object, you register the appropriate Oracle JDBC driver-for example:

DriverManager.registerDriver(
  new oracle.jdbc.OracleDriver()
);

Next, you create a java.sql.Connection object:

Connection myConnection = DriverManager.getConnection(
  "jdbc:oracle:thin:@localhost:1521:ORCL",
  "scott",
  "tiger"
);

To disable auto-commit mode, you then call the setAutoCommit() method of that Connection object. The setAutoCommit() method accepts a Boolean (true or false) parameter; false disables auto-commit mode—which is what we want to do. Here's the statement that disables auto-commit mode for the myConnection Connection object:

myConnection.setAutoCommit(false);

Let's look at an application designed to compare the performance impact of enabling and disabling auto-commit mode.

You can use the AutoCommit.java sample program (see Listing 1) to test my claim that disabling auto-commit mode improves performance. To compile and run this sample yourself, be sure to modify the host name, port number, database service name, schema name, and password as required for your database. As with any Java application that uses the JDBC API for database connectivity, the AutoCommit.java application starts by registering the appropriate JDBC driver (in this case, the Oracle Thin driver) and then creates a Connection object to connect to the database. (You can use either the java.sql.Connection or the oracle.jdbc.OracleConnection object; java.sql.Connection is standard JDBC, whereas oracle.jdbc.OracleConnection is an Oracle JDBC extension and offers more functionality and higher performance.)

After connecting to the database, you create a statement object for the SQL statements (step 3 in Listing 1). From this point on, the program captures data to show the difference in performance between the two modes of operation. The performance table is truncated (see step 4) to clear out any rows that might exist in the table and then auto-commit is enabled (by calling the autoCommit() method on the Connection object (myConnection). The program calls the insertRows() method, which inserts 1,000 rows into the performance table (using a PreparedStatement object comprising the SQL INSERT statement). The program displays the time taken to perform the inserts in milliseconds; auto-commit is then disabled and another 1,000 rows are inserted into the table after the performance table has been truncated to clear out existing data.

Total time for inserting 1000 rows was 5688 milliseconds
Disabling auto-commit mode
Total time for inserting 1000 rows was 3515 milliseconds

As you can see, disabling auto-commit results in a significant performance improvement. Furthermore, disabling auto-commit is generally a good policy, because as a developer you should think about your application's business logic and decide how transactions should be demarcated. Leaving auto-commit enabled is also generally not a good policy because many other JDBC performance-tuning techniques, including update batching (which you'll learn about in the next section), require that auto-commit be disabled.

Using Update Batching to Improve Performance

By default, your application sends one SQL statement at a time to the database. In a production environment, an application typically runs on a different computer from the one on which the database is running, and your application sends each SQL statement over a network to the database. The database receives your SQL statement, runs it, and sends any results back across the network to your application. This entire process is known as a round trip.

As you can imagine, a round trip takes a fairly long time, and sending SQL statements to the database one at time typically isn't efficient. Fortunately, you can combine many SQL INSERT, UPDATE, and DELETE statements into a batch that your application can send to the database for execution at one time, thereby reducing the number of round trips and reducing the overall processing time.

Update batching works well when you are sending the same basic SQL INSERT, UPDATE, or DELETE statement to the database. When you add a different SQL statement to a batch, the batch immediately goes to the database for execution and a new batch is created for the different SQL statement. For this reason, you may not want to use update batching if you're sending many different SQL statements to the database. For example, if you send a lot of INSERT statements to the database, batching works extremely well. On the other hand, if you send an INSERT, followed by an UPDATE, followed by a DELETE, you won't get any benefit from update batching.

Oracle JDBC supports two distinct update-batching models:

  • The Java standard model (referred to as standard update batching in the Oracle documentation and referred to as batch updates in the Sun documentation)
  • The Oracle-specific model (referred to as Oracle update batching), which supports Oracle extensions to JDBC

Generally, you'll find that Oracle update batching performs better than standard update batching. However, if portability of your code between Oracle and non-Oracle databases is important, you should use standard update batching. Both standard and Oracle update batching work only for SQL INSERT, UPDATE, and DELETE statements and only for non-LOB (large object) datatypes—you can't insert CLOB, BLOB, BFILE, LONG, or LONG RAW data in batch mode. Batching is automatically disabled when a SQL statement includes one of these datatypes.

The type of object you work with in your code also dictates which of the update batching models you should use. For example, Oracle update batching works only with OraclePreparedStatement objects. An OraclePreparedStatement object is similar to a PreparedStatement object, except that it contains more functionality, to handle the additional features available in an Oracle database. Typically, an OraclePreparedStatement object also offers higher performance.

You cannot mix the two batching models: You can use either Oracle update batching or standard update batching, but you can't use both at the same time. You'll learn how to use each of these batching models in the following sections.

Using Update Batching with Standard JDBC

As mentioned, standard update batching is part of standard JDBC, and you should use this technique when code portability across non-Oracle databases is important. You can use standard update batching to batch SQL statements performed by the following types of objects in your application:

Statement
OracleStatement
PreparedStatement
OraclePreparedStatement
CallableStatement
OracleCallableStatement

PreparedStatement objects are precompiled statements that you can reuse for executing identical SQL statements, with different values embedded as ?. CallableStatements objects are stored procedures in the database.

For details about standard JDBC, see the online Javadocs for the Statement, PreparedStatement, and CallableStatement interfaces.

The Oracle* objects (OracleStatement and so on) are Oracle extensions to JDBC. You can still use standard update batching with these objects. (For additional information about Oracle extensions to JDBC and OracleStatement, OraclePreparedStatement, and OracleCallableStatement, see Oracle9i JDBC Developer's Guide and Reference Release 2 (9.2).

To use standard update batching in your application, follow these five steps:

1. Disable auto-commit mode.
2. Create a statement object.
3. Add SQL statements to your batch.
4. Execute your batch of SQL statements.
5. Commit (or roll back) the entire batch of executed SQL statements.

Step 1: Disable Auto-Commit Mode

Your first step is to disable auto-commit, to prevent each SQL statement contained in any given batch of SQL statements from committing immediately after the batch has been sent. As mentioned earlier, you always want to disable auto-commit for performance reasons anyway. As an added bonus, disabling auto-commit gives you more control if an error occurs while you are executing a batch, because you can choose to either commit or roll back the statements in a batch that have already been executed. If auto-commit mode is enabled, you don't have this level of control, since any statements are automatically committed. (You'll learn more about error handling later in this article.)

To disable auto-commit mode, simply set the Boolean in the Connection object's setAutoCommit() method to false:

myConnection.setAutoCommit(false);

Step 2: Create a Statement Object

Your second step is to create the appropriate statement object by using one of the statement types listed earlier (PreparedStatement, CallableStatement). This example creates a PreparedStatement object (myPrepStatement) that contains an INSERT statement for adding a row to the performance table:

PreparedStatement myPrepStatement =
  myConnection.prepareStatement(
    "INSERT INTO performance " +
    "(int_value, varchar2_value) VALUES (?, ?)"
  );

The question marks (?) are placeholders that indicate where values will be set in the VALUES clause of the INSERT statement. In this example, we'll set values for the int_value and varchar2_value columns of the performance table—as you'll see in the following step.

Step 3: Add SQL Statements to Your Batch

The third step is to add SQL statements to your batch. Depending on the number of SQL statements in your transaction, you can vary the number of statements in your batch. Although you can batch as many statements as you want, you should avoid flooding the database with a massive number of batched statements. A batch containing 5 to 30 SQL statements is optimal; the batch in this example contains 10 statements.

You add a SQL statement to a batch by using your statement object's addBatch() method. When you first call the addBatch() method, a new batch is created and your first SQL statement is added to that batch. To add another SQL statement to your batch, you call addBatch() again. When you've added as many SQL statements as you need to, you then execute the batch; you'll see how to execute a batch shortly, in step 4. After you've executed the batch, if you call addBatch() again, a new batch is created for you and the process repeats until you've executed all of your SQL statements.

To make these ideas concrete, let's take a look at some code that uses a for loop to add statements to a batch. This loop adds 10 INSERT statements to a batch by using the addBatch() method of myPrepStatement and sets the values for the two placeholders in the INSERT statement:

for (int count = 0; count < 10; count++) {
  myPrepStatement.setInt(1, count);
  myPrepStatement.setString(2, "Test" + count);
  myPrepStatement.executeUpdate();
}

The setInt() method assigns an integer to a placeholder, and the setString() method assigns a string to a placeholder. Each of these methods accepts two parameters: the numeric position of the placeholder in myPrepStatement and the value for that placeholder. In the example, the first parameter for setInt() is 1 and corresponds to the first placeholder that specifies the value for the int_value column in the INSERT statement, and the second parameter is the count variable, which sets the placeholder to values between 0 and 9 in the for loop. Similarly, the setString() method sets the second placeholder to the strings from "Test0" to "Test9". At the end of this loop, the batch will contain 10 INSERT statements that will each add a new row to the performance table, for a total of 10 rows.

Step 4: Execute Your Batch of Statements

You execute a batch of statements by using the statement object's executeBatch() method. This method sends your batch of statements to the database; the database then executes each statement in your batch and returns an array of int elements.

The following example calls the executeBatch() method for myPrepStatement, which sends the batch of INSERT statements added in the previous step to the database for execution:

int [] rowsInserted = myPrepStatement.executeBatch();

Each int element in the array returned by executeBatch() stores something different, depending on the type of statement object you use. You'll learn what these int elements represent for the various statement objects later, in the "Handling Batch Errors" section.

Step 5: Commit (or Roll Back) the Batch of Executed Statements

After executing all the statements in the batch, you must ultimately either commit or roll back the batch by calling either the commit() or rollback() method on the connection object. For example, assuming you created a connection object named myConnection, you would commit the entire batch with the following:

myConnection.commit();

This would commit the 10 INSERT statements executed in step 4. Typically, you write the commit in the try{} block of your Java code, and you call the rollback in the catch{} block—the catch block is where you catch exceptions and errors. Committing assumes that the batched statements all execute without any errors. In your own applications, you should handle errors gracefully, as you'll learn in the next section.

Handling Batch Errors

If an error occurs for any of the SQL statements in the batch, a java.sql.BatchUpdateException object will be thrown. You should catch this exception object in your code and then call the getUpdateCounts() method for the BatchUpdateException object.

The getUpdateCounts() method returns an int array that contains the same values as the executeBatch() method described earlier. Typically, you should also perform a rollback in your catch block, to undo any successfully executed SQL statements.

The following code snippet shows how to catch a BatchUpdateException object and call the getUpdateCounts() method for that object. You'll notice that the commit() method is called within the try block and that the rollback() method is called within the catch block, to roll back any successfully executed SQL statements:

try {
  /*
    ... code that executes a batch of SQL statements
    using a statement object ...
  */
  myConnection.commit();
} catch (BatchUpdateException e) {
  int [] updateCounts = e.getUpdateCounts();
  myConnection.rollback();
}

The updateCounts int array returned by the getUpdateCounts() method will contain different values, depending on the type of statement object (prepared or callable) you used in the try block to execute the batch of SQL statements. You'll learn about these different values in the next two sections. You can use these values as an additional check before deciding to roll back the successfully executed statements in your batch.

Array Values with PreparedStatement and OraclePreparedStatement Objects

If you use a PreparedStatement or OraclePreparedStatement object in your try block, then the int elements in the updateCounts array indicate whether all of the SQL statements in the batch executed successfully. Because the JDBC driver cannot ascertain which exact SQL statement in the batch failed, the int element values apply to all the SQL statements in the batch.

If the int elements have the value -2, all the batched SQL statements executed successfully. If they have the value -3, this indicates there was a problem with one or more of the batched statements.

For example, say you had a batch of 10 INSERT statements and the first nine statements succeeded but the tenth failed. If that were the case, then all 10 of the int elements in the array would contain -3, indicating that there was a problem with one or more of the SQL statements. Some of the SQL statements may have executed successfully—you can use the -3 value as an additional check before deciding to roll back any of those successfully executed SQL statements.

Array Values with Other Statement Objects

If you use a Statement, OracleStatement, CallableStatement, or OracleCallableStatement object in your try block, then the int elements in the updateCounts array specify the number of rows affected by each SQL statement in your batch.

For example, say you had a batch of two DELETE statements and that the first DELETE statement removed four rows and the second removed two rows from a table. The updateCounts array would then contain two int elements: The first int would have the value 4 and the second would be 2—the number of rows affected by each DELETE statement in the batch. If, however, the second DELETE statement in the batch failed, then the updateCounts array would contain only one int element. As you can see, when there's an exception, the number of elements in the updateCounts array is short by 1. You can use this fact as an additional check before you decide to roll back any successfully executed SQL statements in your batch.

Using Update Batching with Oracle Extensions

As mentioned previously, Oracle update batching is generally faster than standard JDBC update batching. and also features more functionality. Oracle update batching does have a limitation, though: You can use it only with OraclePreparedStatement objects. So if you are working with code that contains JDBC PreparedStatement objects, you must cast them to OraclePreparedStatement objects before you use Oracle update batching.

Oracle update batching differs from standard update batching in that you first set a batch value—the maximum number of SQL statements allowed in a batch. Instead of using addBatch() or a comparable method to add SQL statements to a batch, you call the executeUpdate() method on your OraclePreparedStatement object, which adds an SQL statement to the batch automatically. When the number of SQL statements in the batch reaches the specified batch value, the batch is automatically sent to the database for execution. In the sample application, the batch value is 10, which means that on the 10th executeUpdate(), a batch of 10 INSERT statements is sent to the database.

The optimal batch size depends on a variety of factors, but a good rule of thumb for optimal results is to set the batch value to between 5 and 30 SQL statements.

The batch-value setting is one reason why Oracle update batching is usually faster than standard update batching. Because the JDBC driver knows how many SQL statements are to be batched at a time, it can make optimizations based on the batch value.

For Oracle update batching, your code should go as follows:

1. Disable auto-commit mode.
2. Set your batch value.
3. Batch your SQL statements.
4. Commit your executed statements.

Step 1. Disable Auto-Commit Mode

First, disable auto-commit by setting the Boolean in the Connection object's setAutoCommit() method to false:

myConnection.setAutoCommit(false);

Step 2: Set Your Batch Value

You can set the batch value for an OracleConnection or an OraclePreparedStatement object. When you set the batch value for an OracleConnection object, any OraclePreparedStatement objects you create with OracleConnection will automatically use that same batch value. If you set a batch value for an OraclePreparedStatement object, then that batch value will override a batch value you previously set for a OracleConnection object. The default batch value for an OracleConnection or OraclePreparedStatement object is 1, which means that your SQL statements are not batched but are sent one at a time to the database.

In the following two sections, you'll see how to set the batch value for an OracleConnection object and an OraclePreparedStatement object, respectively.

Setting the Batch Value for an OracleConnection Object

To set the batch value for an OracleConnection object, you use the setDefaultExecuteBatch() method. This method accepts an int parameter for the batch value, as shown here:

myOracleConnection.setDefaultExecuteBatch(10);

where the myOracleConnection (OracleConnection object) is set to batch up to 10 SQL statements.

You can also cast an existing connection object to an OracleConnection object and then call the setDefaultExecuteBatch() method, as in this example:

((OracleConnection) myConnection).setDefaultExecuteBatch(10);

The myConnection object-a regular Java connection object-is cast to an OracleConnection and then calls the setDefaultExecuteBatch() method.

To check the current batch-value setting in an OracleConnection object, use the getDefaultExecuteBatch() method:

int batchSize = myOracleConnection.getDefaultExecuteBatch();

The getDefaultExecuteBatch() method returns an int value, which in this case will be 10, since we set the batch value for the myOracleConnection object to 10.

Setting the Batch Value for an OraclePreparedStatement Object

You use the setDefaultExecuteBatch() method to set the batch value for an OraclePreparedStatement object. This method accepts an int parameter containing your batch value; this value overrides the batch value for the OracleConnection object.

For example, assume you have the following OraclePreparedStatement object named myOraclePrepStatement, which contains an INSERT statement:

OraclePreparedStatement myOraclePrepStatement =
  (OraclePreparedStatement) myConnection.prepareStatement(
    "INSERT INTO performance " +
    "(int_value, varchar2_value) VALUES (?, ?)"
  );

As you can see, the INSERT statement is the same as that in the standard update batching example: The first placeholder specifies the value for the int_value column, and the second specifies the value for the varchar2_value column. Next, use the setDefaultExecuteBatch() method of myOraclePrepStatement to set the batch value to 20:

myOraclePrepStatement.setExecuteBatch(20);

This batch value overrides the batch value of 10 previously set for myConnection, so you can use myOraclePrepStatement to batch 20 SQL statements before they are sent to the database.

You can check the current batch value for an OraclePreparedStatement object via its getDefaultExecuteBatch() method—for example:

System.out.println("Batch value = " +
  myOraclePrepStatement.getExecuteBatch());

Your next step is to batch your SQL statements.

Step 3: Batch Your SQL Statements

Once you've set the batch value, all you have to do to add a SQL statement to a batch is call the executeUpdate() method for your OraclePreparedStatement object. When the number of SQL statements in your batch reaches the batch value, your batch will be automatically sent to the database for execution.

This example shows a for loop that adds 20 INSERT statements to a batch, setting values for the int_value and varchar2_value columns for each INSERT statement to 0 through 19 and Test0 through Test19, respectively:

for (int count = 0; count < 20; count++) {
  myOraclePrepStatement.setInt(1, count);
  myOraclePrepStatement.setString(2, "Test" + count);
  int rowsInserted = myOraclePrepStatement.executeUpdate();
}

In step 2, we set the batch value for the myOraclePrepStatement object to 20, so when the 20th INSERT statement is added to the batch, the batch automatically goes to the database for execution.

The int value returned by the executeUpdate() method is the number of rows affected by a SQL statement. Because your SQL statements are batched and then sent to the database as one unit, this returned int value will be 0 if the executeUpdate() method simply adds a SQL statement to the batch (because the SQL statement hasn't yet been executed and so hasn't yet affected any rows). If, however, the executeUpdate() method actually sends the whole batch to the database for execution, the returned int value will contain the total number of rows affected by all of the batched SQL statements added together.

So, in the example for loop, the int value returned by executeUpdate() will be 0 for the first 19 iterations of the loop (because the INSERT statements are just added to the batch and aren't actually executed). But on the 20th iteration, the batch is sent to the database and all the statements are executed, so the int value returned will be 20—the number of rows added to the database by the 20 batched INSERT statements.

Step 4: Commit Your Executed Statements

Your fourth step is to call the commit() or rollback() method on your Connection object. For example, if you created a connection object named myConnection, you would commit the entire batch with

myConnection.commit();

Forcing the Execution of a Batch

You can also force the execution of a batch before it is filled. You do this by calling the sendBatch() method. Typically, you'll never need to use this method because even if your batch isn't full, it will still be sent to the database whenever you perform a commit. However, there may be occasions when you want to force the execution of a batch.

Let's look at an example of using sendBatch(). Say you put three INSERT statements in a batch. You could then force the execution of those three statements by calling sendBatch() and using your OraclePreparedStatement object as follows:

int rowsInserted = myOraclePrepStatement.sendBatch();

The sendBatch() method returns an int value that contains the number of rows affected by the SQL statements in your batch. In this example, sendBatch() will return 3, the number of rows inserted by the statements in the batch, assuming each INSERT statement inserts only one row.

There's Always Room for Improvement

Disabling auto-commit mode and using standard or Oracle update batching to improve the performance of Java applications are just two performance-tuning techniques that can make your applications really fly.
